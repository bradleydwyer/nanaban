## Overview

`nanaban` (nanaban) is a standalone Rust CLI tool for Gemini image generation. It provides text-to-image generation, image editing, and model listing via the Gemini REST API. Designed both for direct human use and for programmatic consumption by Claude Code via structured JSON output.

---

## Commands

### `nanabangenerate "prompt"`
Text-to-image generation. Sends a text prompt to the Gemini API and saves the resulting image.

### `nanabanedit image.png "instructions"`
Image editing. Sends an existing image with text instructions. Returns a modified version.

### `nanabanmodels`
Lists available models with short names, full API identifiers, and estimated per-image costs. No API key required.

## Input Modes

Priority order:

1. **Flag:** `nanabangenerate -p "a cat"`
2. **Positional:** `nanabangenerate "a cat"`
3. **File:** `nanabangenerate -p @prompt.txt` (`@` prefix triggers file read)
4. **Stdin:** `echo "a cat" | nanabangenerate` (when stdin is not a TTY)

Error if both positional and `-p` are provided. Error if no prompt and stdin is a TTY.

## Output

### JSON mode (`--json`)

Stdout:
```json
{
  "status": "success",
  "images": [
    {
      "path": "/absolute/path/to/nb_20260317_143022_a1b2c3.png",
      "width": 1024,
      "height": 1024
    }
  ],
  "text": "optional model text response",
  "model": "gemini-3.1-flash-image-preview",
  "model_short": "flash",
  "elapsed_seconds": 3.42,
  "estimated_cost_usd": 0.04
}
```

Error:
```json
{
  "status": "error",
  "error": "API returned 429: rate limit exceeded",
  "model": "gemini-3.1-flash-image-preview",
  "model_short": "flash",
  "elapsed_seconds": 0.31
}
```

### Human mode (default)
- Progress/status to **stderr**
- Final image path to **stdout** (pipeline-friendly)
- Cost estimate to stderr after each generation

### Auto-open
On macOS, opens image via `open` command. Skipped when `--json` is active or `--no-open` is passed.

### Auto filenames
Pattern: `nb_{YYYYMMDD}_{HHMMSS}_{hash8}.png`
Where `hash8` is first 8 hex chars of a prompt-derived hash.

## Gemini REST API

### Endpoint
```
POST https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent?key={API_KEY}
```

### Text-to-image request
```json
{
  "contents": [{"parts": [{"text": "prompt"}]}],
  "generationConfig": {"responseModalities": ["TEXT", "IMAGE"]}
}
```

### Image editing request
```json
{
  "contents": [{
    "parts": [
      {"inlineData": {"mimeType": "image/png", "data": "<base64>"}},
      {"text": "edit instructions"}
    ]
  }],
  "generationConfig": {"responseModalities": ["TEXT", "IMAGE"]}
}
```

Reference images (`--ref`) are additional `inlineData` parts before the text part.

### Response
`candidates[0].content.parts[]` contains:
- `{"text": "..."}` — text response
- `{"inlineData": {"mimeType": "image/png", "data": "<base64>"}}` — generated image

### Error handling
| Status | Action |
|--------|--------|
| 401/403 | Invalid/missing API key |
| 429 | Rate limited, suggest retry |
| 400 | Bad request, forward API message |
| 500/503 | Server error, suggest retry |
| Safety block | Surface `blockReason` to user |

## Implementation Sequence

### Phase 1: Skeleton + models
1. `cargo init`, set up Cargo.toml
2. Define clap CLI structure in `cli.rs`
3. Implement model registry in `models.rs`
4. Wire up `nanabanmodels`
5. Verify: `nanabanmodels` and `nanaban--help`

### Phase 2: API client + text-to-image
1. Implement `config.rs` (API key resolution)
2. Build `GeminiClient` in `client.rs` with `generate()`
3. Implement base64 decode + image save in `output.rs`
4. Auto-filename generation
5. Wire up `nanabangenerate "prompt"` end-to-end
6. Add `--json` output
7. Verify: `nanabangenerate "a banana" --json`

### Phase 3: Input modes + flags
1. Stdin detection and reading
2. `@file` prefix handling for `-p`
3. `--dry-run` with cost estimation
4. `--aspect` and `--size` → `generationConfig`
5. `--output` for custom paths
6. `--no-open` and auto-open on macOS
7. `-v` verbosity levels (stderr logging)

### Phase 4: Image editing + reference images
1. Image file reading + base64 encoding
2. `edit()` method in `client.rs`
3. Wire up `nanabanedit image.png "instructions"`
4. `--ref` for reference images (repeatable, up to 14)
5. Verify: `nanabanedit input.png "add sunglasses" --json`

### Phase 5: Polish
1. Error messages for all failure modes
2. Cost display on stderr after every generation
3. Full verification checklist
4. Update Gemini skill document
